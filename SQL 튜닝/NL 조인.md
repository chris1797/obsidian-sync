# NL 조인 (Nested Loop Join)

가장 직관적인 조인 방식. **Outer 테이블의 행을 하나씩 꺼내, 그 값으로 Inner 테이블을 찾아간다.** 중첩 루프 그대로의 모양.

```
for each row in Outer:
    Inner 인덱스 탐색 → 매칭되는 행 반환
```

## 동작 흐름

1. Outer 테이블에서 한 행을 읽음
2. 조인 키 값으로 Inner 테이블의 **인덱스 수직 탐색** (루트 → 브랜치 → 리프)
3. 리프에서 ROWID로 **테이블 블록 접근**
4. 매칭되는 행을 결과로 보냄
5. Outer 다음 행으로 반복

핵심은 **Inner 쪽에 좋은 인덱스가 있어야 한다**는 점. 인덱스가 없으면 매 행마다 풀 스캔이라 사실상 못 쓴다.

## 메모리 관점

매 행마다 거치는 인덱스 블록·테이블 블록이 전부 [[SGA]] Buffer Cache에 있다. 그래서:

- 블록 하나 읽을 때마다 `cache buffers chains` 래치 획득
- 동시 사용자가 많으면 같은 블록을 여러 세션이 노려서 **래치 경합** 발생
- Logical Read 자체가 CPU 비용 → 행 수에 비례해서 누적

→ **소량이면 문제없지만, 대량으로 가면 SGA 래치 비용이 결정적으로 불리**해진다. 이 지점에서 [[해시 조인]]에게 자리를 내준다.

[[PGA]]는 거의 안 쓴다. 정렬·해시 같은 중간 작업 공간이 필요 없기 때문.

## 언제 유리한가

- **소량 데이터** 조인 (Outer 행 수가 적을 때)
- **Inner 쪽에 선택도 좋은 인덱스**가 존재할 때
- OLTP의 단건·소건 조회 — 실무에서 가장 자주 만나는 패턴

## 언제 불리한가

- 대량 조인 — 래치 경합 + Logical Read 누적
- Inner 인덱스가 없거나 선택도가 나쁠 때
- 이런 상황엔 [[해시 조인]] 또는 [[소트 머지 조인]]이 답.

## 연결되는 개념

- [[해시 조인]] — 대량 조인에서 NL의 약점(SGA 래치 누적)을 PGA로 우회한 방식.
- [[소트 머지 조인]] — 정렬 후 머지로 처리. 해시 조인이 안 될 때의 또 다른 대안.
- [[SGA]] — NL이 일하는 무대. 래치 경합의 진원지.
- [[PGA]] — NL은 거의 사용하지 않음 (대비점).
