# 해시 조인이 NL 조인보다 빠른 이유 — PGA 메모리 관점

대량 데이터 조인에서 해시 조인이 결정적으로 빠른 이유는 결국 **어느 메모리에서 일하느냐**의 차이로 귀결된다. [[PGA]]/[[SGA]]를 알고 보면 "왜?"가 분명해진다.

## 출발점: SGA vs PGA 다시 보기

- [[SGA]] — 인스턴스 공용. 버퍼 캐시·라이브러리 캐시 등. 여러 세션이 동시에 들여다보니까 **래치(latch) 필요**.
- [[PGA]] — 세션 전용. 정렬·해시 작업용. 나만 쓰니까 **동기화 비용 0**.

이 한 줄 차이가 조인 성능을 가른다.

## NL 조인의 비용 구조

```
Outer 한 행 → Inner 인덱스 수직 탐색 → 리프 → ROWID로 테이블 블록
            (이 모든 블록을 SGA Buffer Cache에서 읽음)
```

문제는 인덱스를 잘 타도 **블록 하나 읽을 때마다 `cache buffers chains` 래치를 잡아야 한다**는 점. 동시 사용자가 많으면 경합이 누적된다. Logical Read 자체가 CPU 비용이라, 1만 건 조인이면 수만 번의 SGA 접근이 쌓인다.

→ **행 수에 비례해서 SGA 래치 비용이 누적되는 구조**. 대량으로 가면 결국 한계가 온다.

## 해시 조인의 비용 구조

```
Build : 작은 테이블을 PGA Hash Area에 해시 테이블로 적재 (한 번만)
Probe : 큰 테이블을 읽으면서 해시 함수로 O(1) 매칭
```

빨라지는 이유 세 가지:

1. **해시 테이블이 PGA(내 메모리)에 있음** → 래치 없이 바로 접근.
2. **해시 함수는 O(1)** → B-Tree 탐색 O(log N)보다 빠름.
3. Build 테이블을 **한 번만 읽고** 메모리에 올려두므로, 같은 블록을 반복 방문하지 않음.

NL이 "행마다 SGA 들락날락"이라면, 해시는 "한 번 PGA에 올려두고 빠르게 매칭".

## 비유

- **NL 조인** = 도서관 공용 서가([[SGA]])에서 책을 한 권씩 꺼내 보는 방식. 매번 사서한테 허락받고(래치), 다른 사람이 같은 책 보고 있으면 기다려야 함.
- **해시 조인** = 필요한 책을 내 책상([[PGA]]) 위에 미리 다 가져다 놓고 보는 방식. 한 번 옮기는 비용은 들지만, 그 다음부터는 누구 눈치도 안 봄.

## 그럼 항상 해시 조인이 답인가?

아니다. 결정적 이점이 발휘되는 **조건**이 있다.

- **대량 데이터 조인** — 행이 많을수록 SGA 래치 비용 누적 → NL 불리.
- **Build 테이블이 PGA에 들어갈 만한 크기** — 너무 크면 디스크 temp로 **spill**(one-pass / multi-pass) 되어 오히려 느려진다. 이 임계점이 [[PGA]] 노트에서 말한 "관전 포인트".
- **소량 데이터 + 좋은 인덱스** — 이 경우엔 NL이 더 빠르다. 해시 테이블 만드는 오버헤드가 아까움.

옵티마이저가 통계 정보를 보고 둘 중 유리한 쪽을 선택하는 이유다.

## 한 줄 요약

> NL 조인은 매번 [[SGA]](공용)에서 블록을 읽으며 래치 경합이 누적되지만, 해시 조인은 [[PGA]](전용)에 해시 테이블을 올려놓고 동기화 비용 없이 O(1) 매칭하기 때문에 대량 조인에서 결정적으로 빠르다.

## 연결되는 개념

- [[PGA]] — Hash Area가 해시 조인 Build 단계의 무대.
- [[SGA]] — NL 조인이 매 행마다 들락거리는 곳. 래치 경합의 진원지.
- [[소트 머지 조인]] — 또 다른 대량 조인 전략. 정렬이 PGA Sort Area에서 일어남. 해시 조인이 안 될 때의 대안.
- [[NL 조인]] — 이 노트에서 비교 대상으로 등장. 별도 정리 필요.
