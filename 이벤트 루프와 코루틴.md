> [[FastAPI 동시성 모델]] 이 "한 스레드 안에서 동시성을 만든다" 라고 했는데, **그 안쪽이 어떻게 굴러가는지** 의 메커니즘. `async`/`await` 가 단순한 키워드가 아니라 **양보 약속** 이라는 것을 이해하면 async 의 모든 함정이 한 줄로 설명된다.

### 1. 코루틴은 "아직 실행되지 않은 함수"

`async def` 함수를 호출하면 **함수 본문이 실행되지 않는다**. 대신 `coroutine` 객체가 반환될 뿐.

```python
async def hello():
    print("hi")

hello()              # ❌ 아무것도 안 찍힘. <coroutine object> 만 받음.
asyncio.run(hello()) # ✅ 이때 비로소 실행
```

→ 이게 처음 어색한 부분. Java/Spring 의 메서드 호출은 **즉시 실행** 이 디폴트인데, 파이썬 코루틴은 호출과 실행이 분리된다. 호출은 "실행 계획을 만든 것" 이고, 실제 실행은 **이벤트 루프가 돌려줘야** 일어난다.

본질적으로 generator 의 사촌. `yield` 자리에 `await` 가 들어가고, "값을 내놓는" 게 아니라 **"제어권을 내놓는"** 게 목적인 generator — 그게 코루틴이다.

### 2. `await` 는 "기다림" 이 아니라 "양보 요청"

이 한 줄이 async 의 80%다.

`await some_io()` 는 "I/O 끝날 때까지 잠든다" 가 아니다. 정확히는:

1. 지금 진행 중인 작업이 **완료 가능하지 않으면** (I/O 대기 등),
2. **제어권을 이벤트 루프에 돌려준다**.
3. 루프는 그 사이 **다른 준비된 코루틴** 을 골라 실행한다.
4. 원래 작업이 완료 가능해지면 루프가 **재개 지점부터** 다시 돌린다.

→ 즉 `await` 가 박힌 지점이 **유일하게 다른 코루틴이 끼어들 수 있는 자리**. `await` 사이 구간은 한 코루틴이 **atomic 하게** 점유한다.

이 관점이 박히면, 왜 `async def` 안에서 `time.sleep(1)` 을 부르면 안 되는지가 자명해진다 — `time.sleep` 은 `await` 를 거치지 않으므로 **양보가 없다**. 1초 동안 다른 모든 요청이 멈춘다. 반면 `await asyncio.sleep(1)` 은 즉시 양보하고, 1초 뒤 재개 큐로 돌아온다.

### 3. 이벤트 루프 = 협력적 스케줄러 + I/O 감시자

이벤트 루프가 한 사이클에서 하는 일은 거칠게 두 가지.

**(a) 준비 큐에서 Task 하나 꺼내 다음 `await` 까지 실행**
- 코루틴은 단독으로 스케줄 안 됨. `asyncio.create_task(coro)` 로 감싸야 **Task** (= 스케줄 가능한 단위) 가 된다.
- `asyncio.run(main())` 도 내부에서 main 코루틴을 Task 로 감싸는 과정.

**(b) I/O 대기 중인 Task 들을 OS 에 등록해 두고, 준비된 것을 큐로 옮김**
- 내부적으로 `selector` (리눅스 `epoll`, BSD/Mac `kqueue`) 를 사용.
- "소켓 N 번 읽기 가능해지면 알려달라" 를 OS 에 등록해 두고, 그 사이 다른 Task 를 돌린다.
- 알림 오면 해당 Task 를 준비 큐로 옮겨 다음 사이클에 재개.

→ 이게 **"한 스레드로 만 개의 커넥션을 처리할 수 있는"** 근본 메커니즘. 스레드를 만 개 만들 필요가 없다. OS 의 I/O 멀티플렉싱 위에 얇은 파이썬 레이어를 얹은 것에 가깝다.

→ Future 는 "아직 결과가 안 정해진 비동기 작업의 placeholder". Task 는 Future 의 서브클래스다. 일반적으로 사용자가 Future 를 직접 만들 일은 없고, `await` 가 코루틴/Task/Future 모두를 받는다는 것만 알면 충분.

### 4. 협력적이라는 말의 무게

스레드 기반 동시성은 **선점적** (preemptive) — OS 스케줄러가 임의 지점에서 스레드를 가로챈다. async 동시성은 **협력적** (cooperative) — 코루틴이 **스스로 `await` 로 양보해야만** 다른 코루틴이 돌아간다.

이 차이가 만드는 결과:

| | 스레드 (preemptive) | 코루틴 (cooperative) |
|---|---|---|
| 양보 시점 | OS 가 임의로 가로챔 | `await` 가 있어야 양보 |
| 락 / 동기화 | 거의 항상 필요 | `await` 없는 구간은 자동 atomic |
| 폭주의 영향 | 다른 스레드는 영향 적음 | 한 코루틴이 전체 루프를 막을 수 있음 |

→ "락이 거의 필요 없다" 는 큰 이득. `await` 사이의 동기 코드는 다른 코루틴이 끼어들 수 없으므로, 공유 자료구조를 만질 때 굳이 락을 안 걸어도 된다.

→ 대신 책임이 사용자 쪽으로 온다. **CPU 무거운 동기 루프** 나 **블로킹 라이브러리** 를 `async def` 안에서 그대로 부르면 그 코루틴이 양보를 안 해서, 같은 워커의 모든 요청이 정지한다. [[FastAPI 동시성 모델]] 에서 말한 "가장 흔히 밟는 지뢰" 의 원리가 정확히 이 지점.

### 5. Spring/Java 와의 좌표 맞추기

스프링 개발자가 async 를 처음 만질 때 헷갈리는 매핑:

- `CompletableFuture<T>` ≈ `Future` / `Task` — "미래에 완료될 값" 이라는 개념은 거의 같다.
- `CompletableFuture.thenApply(...)` 의 콜백 체인 ≈ `await` 로 다음 줄을 잇는 것. async/await 는 **콜백을 평범한 코드처럼 쓰게** 해주는 문법 설탕에 가깝다.
- `@Async` 메서드 ≈ `asyncio.create_task` — "지금 안 기다리고 따로 던지기".
- 결정적 차이: JVM 의 `CompletableFuture` 는 보통 **여러 OS 스레드** 위에서 돈다 (ForkJoinPool 등). asyncio 는 **한 스레드 안의 협력**. 그래서 JVM async 는 락이 필요하고, asyncio 는 보통 안 필요하다.

→ [[GIL]] 이 있어서 어차피 한 스레드만 파이썬을 돌릴 수 있는 환경이라, "스레드 여러 개로 콜백 분산" 모델은 파이썬에서 큰 이득이 없다. 협력적 단일 스레드가 더 자연스러운 답이 된 배경.

### 6. Node.js 와의 좌표 맞추기 — 거의 같은 모델

Spring 비유가 *차이* 를 부각하는 용도였다면, Node 비유는 *같음* 을 부각하는 용도. 둘은 보완적이다.

asyncio 와 Node 의 이벤트 루프는 **본질적으로 같은 아키텍처**다. Node 가 libuv 위에 단일 스레드 이벤트 루프를 돌리는 것과, asyncio 가 `selectors` (리눅스 epoll, Mac/BSD kqueue) 위에 도는 것이 사실상 같은 발상 — OS 의 I/O 멀티플렉싱 위에 얇은 런타임을 얹어, 한 스레드로 수천 커넥션을 처리한다. 콜백 → Promise/Future → `async`/`await` 로 진화한 경로까지 두 언어가 닮아 있다.

차이는 디테일에서 갈린다.

- **워커 스레드풀의 비중**: Node 는 **libuv 워커 스레드풀** (기본 4개) 을 디스크 I/O, DNS, crypto 처럼 OS 가 좋은 비동기 API 를 안 주는 작업에 광범위하게 쓴다. asyncio 의 `loop.run_in_executor` 와 [[FastAPI 동시성 모델]] 의 `run_in_threadpool` 도 같은 역할이지만, Python 쪽은 **"동기 라이브러리를 끌어들이는 다리"** 용도가 더 부각된다. 생태계가 처음부터 async 였던 Node 와 달리, Python 은 동기가 본가고 async 가 뒤따라 들어왔으니 자연스럽다.
- **멀티코어 확장**: 멀티프로세스 (Node `cluster`, FastAPI 워커 N개) 가 표준이라는 점은 같다. 하지만 Node 는 V8 자체가 [[GIL]] 같은 게 없어서 `worker_threads` 로 진짜 병렬 스레드도 쓸 수 있다. Python 은 그 길이 막혀 있어 사실상 프로세스가 유일한 답.
- **같은 함정**: "한 콜백/코루틴이 루프 막으면 끝" 은 양쪽 다 동일. Node 의 무거운 `while(true)` 와 Python 의 `time.sleep(1)` 이 같은 죄. 어느 쪽이든 협력적 모델의 대가는 같다.

→ 한 줄로: **asyncio = 파이썬판 Node 이벤트 루프**. 차이는 GIL 과 생태계 출발점이지, 모델 자체가 아니다.

### 의문점 / 더 파볼 것

- `asyncio.create_task` 로 던진 Task 가 예외를 던지면 어떻게 잡히나? **Task 의 예외는 await 하기 전엔 silent**, 이게 또 하나의 함정. → `[[asyncio.gather]]` 또는 TaskGroup 노트로 따로 정리할 가치.
- `await` 는 양보지만, 어떤 `await` 는 즉시 완료 가능해서 양보가 실제로 일어나지 않을 수도 있다 (예: 이미 완료된 Future). 이게 성능 측면에선 좋지만, "내 코루틴이 분명 양보했을 줄 알았는데" 같은 미스를 만든다.
- `loop.run_in_executor(...)` / [[FastAPI 동시성 모델]] 의 `run_in_threadpool` — 동기 함수를 async 세계로 끌어들이는 다리. 이건 별도 노트로 더 깊이 갈 만한 주제.
- async generator (`async for`) 의 작동 — generator 안에서 또 `await` 하는 구조. 스트리밍 응답에서 쓰임.
- [[AsyncSession]] 의 lazy loading 차단이 결국 "lazy 트리거 지점에 `await` 가 없어서" 인데, 이 노트의 2번 절을 알고 나면 그 에러 메시지(`MissingGreenlet`)의 의미가 즉시 보인다.
