### **N+1 문제란?**

: 엔티티 1개를 조회할 때 연관된 자식 엔티티를 접근할 때마다 쿼리가 1번씩 추가로 발생하는 문제

- 예제 코드
```java
// 1번 쿼리 
List<Post> posts = postRepository.findAll(); 

// N번 쿼리 (게시글마다) → 총 1+N번의 쿼리가 날아감. 
for (Post post : posts) {
	System.out.println(post.getComments()); 
}
```

- join fetch 를 사용하면 해결된다. 하지만 근본적인 해결법은 아니다.
```java
@Query("SELECT p FROM Post p JOIN FETCH p.comments") List<Post> findAllWithComments();
```

### 📌 1. `EAGER`는 항상 강제된다 → 내가 원치 않아도 JOIN 발생

- 즉시 로딩은 **조회하는 순간** 자동으로 연관된 엔티티를 로딩함.
  이는 **원하는 쿼리를 제어할 수 없다는 문제**로 이어짐.
- 이는 의도치 않게 불필요한 컬렉션까지 조회해서 성능 낭비를 초래함.

### 📌 2. 여러 컬렉션을 EAGER로 가져오면 `JOIN FETCH` 불가

→ Hibernate는 **한 번에 둘 이상의 컬렉션 fetch join 불가**  
→ 아래 쿼리 불가능:
```sql
SELECT p FROM Post p
JOIN FETCH p.comments
JOIN FETCH p.likes
```

→ 왜냐면 결과 row가 곱집합(`cartesian product`)처럼 늘어나서 
   **중복 데이터와 메모리 폭발** 발생

### 📌 3. 페이징이 깨진다 (중요)

컬렉션을 fetch join 할 경우,

```sql
SELECT p FROM Post p JOIN FETCH p.comments
```

- 결과 row 수가 `Post`가 아니라 `Comment` 수만큼 늘어나므로
- `Pageable`과 함께 사용하면 **결과가 부정확해짐**

### 📌 4. 즉시 로딩은 유지보수가 어렵고, 예측이 어렵다.

- 어떤 쿼리든 간에 무조건 연관된 컬렉션까지 끌고 오므로
  **성능 분석, 쿼리 최적화가 매우 힘들어짐**
- `JPA`가 알아서 로딩할 거라 생각하지만, **실제로는 의도하지 않은 복잡한 쿼리가 생기기 마련**
