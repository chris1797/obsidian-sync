> Global Interpreter Lock — 한 프로세스 안에서 한 시점에 단 하나의 스레드만 Python 바이트코드를 실행하게 만드는 락.

CPython 구현체에 한정된 이야기 (PyPy 도 있음, Jython·IronPython 은 없음). Python 3.13 에서 free-threaded 빌드가 실험적으로 도입됨.

### 왜 있는가

CPython 의 메모리 관리가 **레퍼런스 카운팅** 기반이라서. 객체마다 `ob_refcnt` 를 증감하는 연산이 잔뜩 일어나는데, 이걸 멀티스레드에서 락 없이 만지면 레이스 컨디션 → 메모리 누수 / 이중 해제.

→ 모든 객체마다 락을 거는 대신, **인터프리터 전체에 락 하나** 거는 게 단순했다. 단일 스레드 환경에서는 성능 비용이 거의 0 — 이게 GIL 이 오래 살아남은 이유.

### 실질적 영향

| 작업 종류 | 멀티스레드 효과 |
|---|---|
| **CPU-bound** (계산 위주) | ❌ 없음. 스레드 N 개 띄워도 GIL 때문에 한 번에 하나만 돈다. |
| **I/O-bound** (네트워크, 디스크) | ✅ 있음. I/O 시스템콜에 들어가면 GIL 을 놓는다. 다른 스레드가 그 사이 일한다. |
| **C 확장** (numpy 내부 등) | ✅ 있음. 무거운 C 코드 구간은 명시적으로 GIL 을 놓을 수 있음. |

→ 즉 GIL 은 "파이썬 스레드는 쓸모없다" 가 아니라, **"CPU 병렬엔 스레드 말고 프로세스를 써라"** 라는 제약.

### [[FastAPI 동시성 모델]] 과의 연결

FastAPI 가 운영에서 워커 프로세스를 N 개 띄우는 이유가 정확히 이것. 한 프로세스 안에서 스레드를 늘려도 CPU 코어를 다 못 쓰니까, 어쩔 수 없이 프로세스로 확장한다.

흥미로운 점: asyncio 자체는 **단일 스레드 협력적 동시성**이라 GIL 영향을 거의 안 받는다. 어차피 한 스레드 안에서만 코루틴이 도니까. → async 모델은 GIL 과 궁합이 좋다 — 멀티스레드 동기화 비용 없이 동시성을 얻는다. (협력적 양보의 메커니즘은 [[이벤트 루프와 코루틴]] 참고.)

### JVM 과의 차이

Spring Boot 가 한 프로세스로 코어를 전부 쓸 수 있는 건 JVM 에 GIL 같은 게 없기 때문. JVM 의 스레드는 OS 스레드 위에 1:1 매핑되고 진짜 병렬로 돈다.

→ Spring 은 스레드풀 크기로 동시성을 조절하고, Python 은 프로세스 개수로 조절한다. 같은 "스케일링" 이라는 단어 안에 다른 단위가 들어 있는 셈.

### 의문점 / 더 파볼 것

- Python 3.13 의 free-threaded 빌드는 실제로 어떻게 레퍼런스 카운팅 문제를 푸는가? (per-object lock? biased ref counting?)
- `multiprocessing` 으로 우회할 때 IPC 비용은 어떻게 잡나 (shared memory, queue)
- async + 멀티프로세싱 (워커 N 개 × 각자의 이벤트 루프) 의 표준 구성
